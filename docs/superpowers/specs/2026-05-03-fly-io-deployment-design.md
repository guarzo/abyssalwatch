# Fly.io Deployment — Design

**Date:** 2026-05-03
**Status:** Approved (brainstorming)
**Scope:** Hobby-tier deployment of AbyssalWatch to Fly.io with Supabase Postgres and a custom domain.

## Goals

- Get AbyssalWatch running on Fly.io as a single-region, single-machine deployment.
- Use Supabase as a managed Postgres provider.
- Serve the app on a user-owned custom domain over HTTPS.
- Keep cost low (shared-cpu-1x, auto-stop when idle).
- Brief downtime during deploys is acceptable.

## Non-goals

- Multi-region, HA, or autoscaling.
- Multi-node BEAM clustering.
- Managed Postgres on Fly itself.
- Email delivery / Mailer adapter changes.

## Architecture

Single Fly app, one shared-cpu-1x machine (~512MB–1GB RAM) in a region close to the user (e.g. `iad`). Phoenix release built via `mix phx.gen.release --docker` and deployed as a Docker image to Fly. TLS is terminated by Fly. The custom domain is attached via `fly certs` with A/AAAA (or CNAME) DNS records.

Postgres is hosted at Supabase (free tier), reached via the **transaction pooler** connection string (port 6543) over the public internet. Supabase region picked to match the Fly region (e.g. Supabase `us-east-1` ↔ Fly `iad`) to minimize latency. Pooler is the right fit for a single small machine with ≤ ~10 connections and matches free-tier connection limits.

Background processes (`Watchlists.Monitor` GenServer, `Mutamarket.Cache` ETS, `Mutamarket.RateLimiter`) run in-process on the single machine. State is in-memory and resets on deploy — acceptable: monitor results are re-derived from DB and cache simply re-warms.

External integrations: EVE SSO + ESI (OAuth2 redirect to the custom domain), Mutamarket API (cached + rate-limited), Discord webhooks for notifications.

## Trade-offs

- Supabase vs Fly Postgres: ~20–60ms extra DB latency, but managed backups, a SQL UI, and no DB ops on the user.
- Single machine: brief downtime (~10–30s) on deploys; no HA. Acceptable for hobby.
- Auto-stop: machine sleeps when idle to save money; first request after sleep cold-starts in ~1–2s.

## Files added / changed

### Added (generated)
- `Dockerfile` — multi-stage build (Elixir/Erlang builder → Debian slim runtime). Generated by `mix phx.gen.release --docker`.
- `.dockerignore` — generated by the same task.
- `rel/overlays/bin/server` and `rel/overlays/bin/migrate` — release scripts; `server` sets `PHX_SERVER=true`, `migrate` runs Ecto migrations.
- `fly.toml` — generated by `fly launch --no-deploy`. Single app, one primary region, internal port 4000, `release_command = "/app/bin/migrate"`, auto stop/start machines, healthcheck on `/`.

### Changed
- `config/runtime.exs` — uncomment `ssl: true` on `Abyssalwatch.Repo` and add `ssl_opts: [verify: :verify_none]` (Supabase uses a non-public CA; standard pattern for Supabase + Ecto).

### Not changed
- Ash domains, LiveViews, `Watchlists.Monitor`, `Mutamarket.Cache`, `Mutamarket.RateLimiter` — all work as-is on a single-machine release.

## Secrets (set via `fly secrets set`)

- `SECRET_KEY_BASE` — generated by `mix phx.gen.secret`.
- `DATABASE_URL` — Supabase transaction-pooler URL (port 6543).
- `PHX_HOST` — the custom domain (e.g. `yourdomain.com`).
- `EVE_CLIENT_ID`, `EVE_CLIENT_SECRET` — from CCP developer portal.
- `EVE_CALLBACK_URL` — `https://<PHX_HOST>/auth/eve/callback`.
- `MUTAMARKET_API_KEY` — Mutamarket key.
- `DISCORD_WEBHOOK_URL` — Discord watchlist notifications.

`DNS_CLUSTER_QUERY` is intentionally unset (no clustering).

## Rollout order

1. Create Supabase project; capture transaction-pooler connection string.
2. `mix phx.gen.release --docker`; commit generated files.
3. `fly launch --no-deploy`; commit `fly.toml`.
4. `fly secrets set ...` for everything in the Secrets section.
5. `fly deploy` — first deploy runs migrations via `release_command`.
6. `fly certs add yourdomain.com` — Fly returns DNS records to set.
7. Add DNS records at the domain registrar; wait for cert issuance.
8. Update EVE SSO callback URL in CCP developer portal to `https://yourdomain.com/auth/eve/callback`.
9. Smoke-test: login flow, ESI fittings fetch, watchlist match → Discord webhook, Mutamarket search.

## Known limitations (accepted)

- Brief downtime during deploys (single machine, rolling deploy).
- `Watchlists.Monitor` in-memory state resets on deploy — monitor results re-derived from DB.
- ETS cache cold on every deploy; warms on first requests.
- Auto-stop adds ~1–2s cold-start latency to the first request after idle.
- No HA: if Supabase or the single Fly machine is down, the app is down.

## Verification

After step 9:
- `https://yourdomain.com/` loads with valid TLS.
- EVE SSO login completes and lands back on the app.
- `fly logs` shows migrations ran on first deploy.
- `fly status` shows one healthy machine.
- A test watchlist match posts to the Discord webhook.
